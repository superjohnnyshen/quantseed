# 个人量化策略标准化投研上线 SOP V3.1

_面向个人量化投资者的通用流程框架。适用任何量化策略的全生命周期管理：立项 → 数据 → 因子 → 回测 → 调参 → 盲测 → 模拟盘 → 实盘 → 迭代。_

> **关于本 SOP：** 这套流程不是凭空设计的，也不是机构方法论——而是在一个真实运行的个人量化策略项目（尾盘 T+1 隔夜策略，2018-2026 回测夏普 1.37，OOS 夏普 1.81）中反复踩坑后沉淀下来的。每条规则背后都有具体的失败案例支撑——这些案例也写在了对应章节中。

### 30 秒速览

```
第一阶段：立项     → 想清楚再动手，别事后改标准
第二阶段：数据     → 一个数据源用到底，别混用，别用前复权
第三阶段：因子     → 信 IC 不信直觉，好因子全气候有效
第四阶段：回测     → 引擎必须 100% 匹配实盘，缺一个成本项就白跑
第五阶段：风控     → 参数只调一次，调完永久冻结。消融必做
第六阶段：盲测     → 样本外只跑一次，只看全局指标，看完不许回头改
第七阶段：上线     → 模拟盘→小资金→梯度扩容，不许一步到位
第八阶段：迭代     → 大版本重走全流程，小版本从第五阶段起
```

### 不适用场景

本 SOP 基于 **个人量化投资者的 A 股日频多因子策略** 经验总结，以下场景不完全适用：

| 场景 | 为什么不适用 |
|------|-------------|
| 高频/日内交易 | 关注点不同（延迟、冲击成本、成交率），建议参考市场微观结构文献 |
| 纯单因子策略 | 没有模型训练环节，第三~五阶段可大幅简化 |
| 加密货币 | 24×7 交易、无涨跌停、费率结构完全不同，第二阶段数据规则需重写 |
| 纯基本面/事件驱动 | 调仓频率低（月/季度），回测验证周期长，第七阶段试运行时间需翻倍到数月 |

如果你在这些场景下工作，可以保留本 SOP 的**阶段框架和核心原则**，但各阶段的**具体规则和阈值**需要根据你的市场重新校准。

## 选择你的路径

这份 SOP 覆盖了从立项到实盘的全流程，但**不是每个人都必须走完 8 个阶段**。根据你的策略类型和目标，选择一条路径：

| 路径 | 适用场景 | 走哪些阶段 | 大约耗时 |
|------|---------|-----------|----------|
| **轻量路径** | 规则型策略（均线、动量、纯排序），学习/验证想法，不涉及 ML 模型 | ①立项 → ②数据 → ④回测 → ⑥盲测（4 个阶段） | 1-2 周 |
| **完整路径** | ML 模型策略（LGBM 等），准备上实盘 | 全部 8 个阶段 | 1-3 个月 |

> **怎么选？** 如果你只是"用均线金叉买、死叉卖，跑个回测看看"，走轻量路径就够了。如果你在训练 LGBM 模型做多因子选股，走完整路径。**先跑通轻量路径，再考虑完整路径——不要一上来就追求完美。**

### 各阶段耗时参考

| 阶段 | 轻量路径 | 完整路径 |
|------|---------|----------|
| ① 立项 | 半天 | 1 天 |
| ② 数据 | 1 天 | 3-5 天 |
| ③ 因子/模型 | 跳过 | 1-2 周 |
| ④ 回测 | 1-2 天 | 3-5 天 |
| ⑤ 风控调参 | 跳过 | 2-3 天 |
| ⑥ 盲测 | 半天 | 半天 |
| ⑦ 模拟盘 | 1 周 | 2 周 |
| ⑧ 小资金实盘 | 2 周 | 1 个月 |

> 以上是个人利用业余时间开发的估算，全职开发可缩短到 1/3 ~ 1/2。

## 术语速查表

SOP 中会用到一些量化术语，首次出现时会有解释，这里提供快速索引：

| 术语 | 全称 | 大白话解释 |
|------|------|-----------|
| IC | Information Coefficient | 因子打分和实际收益的相关性，简单说就是"这个因子预测得准不准"。正数说明预测方向对，越大越好 |
| RankIC | Rank Information Coefficient | 和 IC 类似，但只看排名不看数值，更稳健 |
| IR / ICIR | Information Ratio | IC 均值除以 IC 标准差，衡量"预测稳定性"。值越高说明因子的预测能力越稳定 |
| 夏普 / Sharpe | Sharpe Ratio | 每承担一单位风险能赚多少超额收益，简单说就是"性价比"。>1 算及格，>1.5 算优秀 |
| 最大回撤 | Max Drawdown | 净值从最高点跌到最低点的最大幅度，衡量"最惨的时候亏多少" |
| OOS | Out-of-Sample | 样本外数据，即模型从未见过的数据，用来检验策略是否真的有效 |
| WF | Walk Forward | 一种滚动回测方法，用历史数据模拟真实交易中的"逐步推进" |
| Bin / 分层 | Binning | 把股票按因子打分分成 N 组，看每组收益是否有规律 |
| AUC | Area Under Curve | 模型区分好坏股票的能力，0.5=瞎猜，0.7=还行，0.8=很好 |
| 过拟合 | Overfitting | 模型在历史数据上表现完美，但一遇到新数据就崩——就像考试前背了答案但没学会知识 |

## 核心原则

| 原则 | 说明 |
|---|---|
| 流程不可逆 | 严格按阶段顺序，上一阶段验收不通过，禁止进入下一阶段 |
| 数据隔离 | 训练集 / 验证集 / 样本外集物理隔离，样本外封存至盲测阶段 |
| 参数冻结 | 所有参数仅允许在训练集内单次调整，调整完成永久冻结 |
| 规则对齐 | 回测规则必须 100% 匹配实盘操作，差异必须先修正引擎 |
| 全程留痕 | 所有操作、调参、回测结果书面记录，版本化管理 |
| 回测前自动校验 | 每次回测前必须通过 look-ahead / 数据对齐 / 特征完整性 三项自动检查 |
| 消融必做 | 任何新增特征/过滤/参数，必须做消融测试验证独立贡献 |
| 数据快照 | 每个大版本的数据集冻结快照，禁止回测期间增量更新 |

> **案例：参数冻结为什么是红线。** 在 LGBM 模型的 ATR 止盈阈值调优中，训练集上从 0.5 调到 0.58，夏普从 1.37 升到 1.81。当时很想继续调到 0.62——训练集夏普还能到 1.95。但这是过拟合：一个参数调三次以上，本质上是在拟合噪声而非信号。所以 0.58 冻结后，即使看到更好的参数也不再改。训练集上多赚的每一分钱，样本外都可能加倍还回去。

> **案例：数据隔离的重要性。** 项目曾用 Tushare 前复权（qfq）数据做回测，后来切换到 QMT 后复权（hfq）数据。两个数据源的复权因子不一致，导致同一策略在 Tushare 回测夏普 1.8，换成 QMT 数据后模拟盘跑三天就亏 2%。原因排查了两天——不是策略失效，是历史信号全是复权口径不一致导致的幻觉。如果当时样本外数据做了封存和独立校验，这个问题三分钟就能定位。

## 第一阶段：策略立项与目标定义

**目的：** 防止无目标开发，杜绝事后根据回测改标准。

**必须明确：**

| 项目 | 内容 |
|---|---|
| 策略名称 | 唯一标识 |
| 交易逻辑 | 一句话描述核心逻辑 |
| 交易周期 | 日内 / 隔夜 / 周度 / 月度 |
| 持仓时长 | 最长持有天数 |
| 资产范围 | 股票池范围（全市场 / 行业 / 指数成分） |
| 目标年化 | ≥ ___% |
| 目标夏普 | ≥ ___ |
| 最大回撤红线 | ≤ ___% |
| 单年最大亏损 | ≤ ___% |
| 信号区分度门槛 | Bin9 胜率 / IC 均值 / RankIC |
| 单票仓位上限 | ≤ ___% |
| 总仓位上限 | ≤ ___% |
| 手续费 | 佣金、印花税、滑点预设值 |

**数据集划分（完成后封存样本外）：**

| 数据集 | 时间区间 | 用途 |
|---|---|---|
| 训练集 | 自主定义 | 模型训练 + 参数调优 |
| 验证集 | 训练集内划分 | 超参选择（如 Walk Forward） |
| 样本外集 | 训练集之外 | 全阶段封存，仅第六阶段盲测 |

> 数据区间选择必须给出合理业务依据（如市场制度变迁、流动性变化等）。

**交付物：** 《策略立项单》+《数据集划分与封存记录》

**禁止：**
- 提前查看样本外集
- 模糊定义目标，事后修改

> **案例：立项阶段能省掉的弯路。** 在一个项目中，团队曾同时评估三个新方向：可转债双低策略、跨境 ETF 日内策略、ETF 行业轮动策略。因为没有在立项阶段严格评估"信号源是否与现有策略重叠"，ETF 行业轮动全链路搭建完成后才发现——LGBM 模型的 12 个特征（动量、估值、波动率）已经覆盖了行业轮动的核心信号，再做一轮是重复劳动。跨境 ETF 方向则反过来：逻辑成立，但搭建分钟级回测引擎和实时信号执行需要数周工程投入，立项时低估了基础设施成本。这两个教训说明：立项阶段的"信号源冲突检查"和"工程代价评估"不是走过场，是真的能省几周时间。

## 第二阶段：数据基建与校验

**目的：** 建立统一、干净、可追溯的数据流水线。一次建设，永久复用。

**标准流程：**

1. 选定数据源，固定接口与字段名
2. 固定复权口径与交易日历，全程不可变更
3. 自动化清洗流水线：
   - 剔除 ST / *ST / 退市 / 长期停牌个股
   - 标记不可交易状态（涨跌停封死、停牌）
   - 处理异常值与跳空
4. 列式存储（建议 Feather / Parquet），按需分区
5. 编写每日增量更新脚本：自动拉取 → 清洗 → 校验 → 入库
6. 数据异常自动告警

**交付物：**
- 《数据清洗规则》
- 全量标准化数据集
- 每日增量更新脚本

**禁止：**
- 混用不同数据源或复权口径
- 手动修改原始数据

> **案例：数据源切换的代价。** 项目最初使用 Tushare 前复权（qfq）数据，后来想切换到 QMT 后复权（hfq）。两个数据源的复权因子基准不同，导致同一策略回测结果差异巨大——Tushare 回测夏普 1.8，QMT 数据上模拟盘跑三天就亏 2%。排查了两天才定位到是复权口径不一致，而非策略失效。qffq 还有一个陷阱：每次增量更新后，历史价格会随复权因子重新计算而漂移，导致昨天的回测结果和今天的不一样。最终选择 hfq 并单独存储 adj_factor 列，确保回测结果可复现。

### 二-A：数据质量标准（V4.2 审计新增）

以下规则已通过 V4.2 全量代码审计验证，写入 SOP 永久执行。

#### 1. 幸存者偏差（🔴 致命）

| 规则 | 说明 |
|------|------|
| 退市股历史数据必须保留 | 退市只在 `delist_date` 之后标记，之前正常交易日的数据不可删除 |
| ST 标记时间序列化 | 不能用当前快照的 `is_st` 静态值覆盖整个历史；需按 `trade_date` 存 ST 状态变化 |
| 持仓退市处理 | 退市日必须强制平仓，估值按最后成交价而非买入成本 |

**检查点：** 股票基础信息表每只股票只存一条记录 = BUG；ST 状态从名称快照推导而非时间序列存储 = BUG。

> **真实发现：** 在 V4.2 代码审计中，发现 `stock_basic` 表每只股票只存了一条记录，没有按时间序列存储 ST 状态变化。这意味着如果一只股票 2018 年被 ST、2020 年摘帽，代码会用当前快照的状态覆盖整个历史——2018-2020 年间的正常交易日也会被错误标记为 ST 并排除。这在回测中会系统性低估收益（因为排除了后来摘帽的好股票）。

#### 2. 复权口径（🟠 严重）

| 规则 | 说明 |
|------|------|
| 使用后复权（hfq） | 前复权（qfq）导致每次增量更新后历史价格漂移，不可用于回测 |
| 存储 `adj_factor` | 单独存复权因子列，可随时从前复权/后复权/不复权互相转换 |
| 禁止混用复权口径 | 同一只股票在整个历史期内必须使用一致复权方式 |

**检查点：** 数据加载代码使用前复权（qfq）= BUG；增量更新与历史数据复权基准不一致 = BUG。

> **真实发现：** V4.2 审计中发现 `data.py` 使用了前复权（qfq），导致每次增量更新后历史价格漂移。同一个策略在周一和周三跑回测，结果可以差 0.5 个夏普点——因为周二增量更新后，前复权因子重新计算，所有历史收盘价都变了。

#### 3. 停牌处理（🟠 严重）

| 规则 | 说明 |
|------|------|
| 停牌日标记 `is_suspended` | 用于回测/模拟盘跳过该日交易 |
| 停牌期间指标失真 | `.shift(N)` 不会跳过停牌日，需用交易日计数而非自然日 |
| 停牌持仓不可冻结 | 持仓价值不得冻结在买入成本；停牌期间按最后收盘价估值 |

**检查点：** 回测引擎中停牌持仓价值冻结在买入成本价 = BUG。

> **真实发现：** V4.2 审计中发现回测引擎在停牌期间将持仓价值冻结在买入成本价。这意味着如果一只股票买入后停牌半年，期间下跌了 30%，回测净值会显示一条平滑的直线——而真实账户已经亏了 30%。这会导致回测大幅低估最大回撤。

#### 4. 费率标准（🔴 致命）

| 规则 | 说明 |
|------|------|
| 佣金最低 5 元 | `max(amount × rate, 5.0)`，不可忽略 |
| 佣金费率 | 万 1~万 1.5（可谈），代码默认万 1 |
| 印花税仅卖出 | 万 5（0.05%），刚性。ETF/可转债免征 |
| 滑点分层 | 按成交额分三档：>5亿/万三、>5000万/万五、<5000万/千一 |

**检查点：** 回测和模拟盘佣金缺 `max(…, 5.0)` = BUG。

> **真实发现：** V4.2 审计中发现回测引擎和模拟盘引擎的佣金计算都漏掉了"最低 5 元"规则。对于 5 万元的小资金账户，单笔交易金额约 5000 元，万 1 佣金仅 0.5 元，但实际券商收取 5 元——实际佣金是回测的 10 倍。这个 bug 导致回测年化收益虚高约 1.5-2 个百分点。

#### 5. 模拟盘与回测对齐（🟠 严重）

| 规则 | 说明 |
|------|------|
| 仓位计算公式一致 | `paper_engine` 和 `backtest_engine` 的 ATR 仓位公式必须逐项对齐 |
| `stop_mult` 因子 | 尾盘 T+1=1.2，老鼠潜伏=1.5，不可缺失 |
| 定期对比回测 vs 模拟盘差异 | 每月对比收益率、回撤、换手率偏差 |

**检查点：** `paper_engine` 和 `backtest_engine` 的仓位计算公式逐项对比，缺 `stop_mult` 因子 = BUG。

> **真实发现：** V4.2 审计中发现模拟盘引擎的仓位计算公式缺少 `stop_mult` 因子（尾盘 T+1 策略应为 1.2，老鼠潜伏策略应为 1.5）。这意味着回测中单票仓位为 10%，模拟盘实际下单可能是 12%——回测和模拟盘跑的不是同一个策略。这个 bug 在模拟盘跑了近一周后才被发现，期间所有绩效数据都不具备参考价值。

#### 6. 异常值与缺失值

| 规则 | 说明 |
|------|------|
| 涨跌幅缩尾 | 主板 ±9.5%、科创板 ±19.5% 截断 |
| PE/PB 截断 | 99 分位以上截断，负值置 NaN |
| PE/PB 前向填充 | 财报发布间隔中 `ffill()` 填充，避免无故排除股票 |
| 成交量异常 | vol > vol_ma5 × 5 的暴涨暴跌日标记 |

#### 7. 数据源一致性

| 规则 | 说明 |
|------|------|
| 单一数据源 | 全项目统一使用一个数据源（如 Tushare Pro、QMT xtdata），禁止混用多个来源 |
| 禁止 API 代理 | 直连数据 API，清除 HTTP_PROXY 等代理环境变量（代理可能返回过期数据或 502） |
| Token 统一管理 | 通过统一配置文件管理 API Token，禁止在代码中硬编码 |

## 第三阶段：因子工程与模型研发

**目的：** 开发稳定预测能力的因子与模型，严格通过稳定性检验。

**单因子检验标准：**

| 检验项 | 方法 | 通过标准 |
|---|---|---|
| 分组单调性 | 全训练集分 10 Bin，统计分层收益 | 单调递增 / 递减 |
| IC 均值 | 日频 IC 序列均值 | ≥ 0.02 |
| ICIR | IC 均值 / IC 标准差 | ≥ 0.3 |
| 分年度稳定性 | 每年独立计算 IC/RankIC | 各年方向一致 |
| 极端行情表现 | 单独检验熊市 / 震荡市 | 不出现大幅反向 |

剔除规则：IC 长期为负、分层杂乱、年度波动剧烈的因子直接丢弃。

> **案例：不要因为一个因子"听起来有道理"就留着。** 在一个项目中，moneyflow（资金流）因子在直觉上非常有吸引力——主力资金流入应该预示上涨。但全量 RankIC 计算结果是 **-0.017**，不仅不预测，还轻微反向。在 LGBM 特征重要性排名中，它也排在所有 12 个特征的倒数第一。类似的，行业中性化处理被寄予厚望（消除行业偏差），但实际 ΔIC 仅为 **0.0003**——几乎为零增益，却增加了代码复杂度。这两个案例说明：**直觉是因子工程的敌人，IC 才是唯一的裁判。**

> **案例：好因子长什么样。** ma_dev（均线偏离）因子在 2018-2026 全区间 RankIC 为 **0.31**，分年度检验：牛市 0.33、熊市 0.36、震荡市 0.31——全气候有效，方向一致。这是通过检验的因子应该有的样子。另一个周频版本 week_ma_dev 的 IC 高达 **0.66**，说明周频信号比日频信号更干净。这些数字不是选出来的——是先算 IC，再决定用不用。

**模型训练标准：**

1. 目标标签与策略交易逻辑一致（分类用二值，回归用连续值）
2. 严格时间序列分割——使用 Walk Forward 滚动窗口或时间序列交叉验证，禁止随机 shuffle
3. 超参搜索限定范围，搜索轮次 ≤ 200（Optuna 等贝叶斯优化在 100-200 轮内通常已收敛。超过 200 轮仍在"提升"的，大概率是在拟合噪声而非信号。实际经验：LGBM 的 Optuna 调参在 150 轮后验证集 AUC 提升曲线已完全走平。）
4. 评估指标优先看稳健性指标（IC 均值、ICIR、Bin 单调性），而非单区间收益
5. 随机打乱标签对照测试——确认模型不是随机拟合

**交付物：**
- 《单因子检验报告》
- 《模型训练报告》（含特征重要性排序）
- 固化模型文件 + 特征清单

**禁止：**
- 用样本外数据训练
- 无限制扩大超参搜索空间
- 为提高收益随意堆因子

> **案例：因子堆砌的陷阱。** 在一个项目中，团队尝试将 LGBM 特征从 12 个扩展到 14 个（V5 计划）。新增特征后 AUC 仅提升 **+0.0089**，几乎可以忽略。但新增特征带来的代码复杂度、数据维护成本、以及潜在的过拟合风险是实实在在的。最终决定不切换，V4 继续运行。这验证了一个原则：**特征数量不是越多越好，增量收益必须有明显改善才值得。**

> **案例：择时层的代理指标陷阱。** 在一个项目中，团队尝试用沪深 300 指数作为市场择时信号，构建了一个择时层。训练集上表现良好，但 OOS 期间年化收益从 12.3% 降到 2.2%，夏普从 0.96 降到 0.27。根因是 CSI300 只是市场的一个代理指标，不能真正代表全市场涨跌家数。当代理指标本身失真时，择时信号就是噪音。结论：**择时策略的瓶颈通常在代理指标的选择，而非模型本身。**

## 第四阶段：回测引擎开发与交易规则对齐

**目的：** 回测规则 100% 匹配实盘，消除引擎差异导致的绩效失真。

**引擎必须包含的模块：**

| 模块 | 必须实现 |
|---|---|
| 选股 | 严格按规定时点可用的数据生成信号 |
| 买入撮合 | 买入时点、价格逻辑与实盘一致 |
| 卖出撮合 | 卖出触发条件、成交价格与实盘一致 |
| 不可交易 | 涨跌停 / 停牌当日跳过，顺延逻辑明确 |
| 仓位管理 | 单票上限、总仓位上限、现金 / 债券管理 |
| 交易成本 | 佣金、印花税、滑点（建议分档）全部写入 |

**回测双模式：**

| 模式 | 用途 | 能否用于上线决策 |
|---|---|---|
| 快速模式 | 淘汰垃圾模型 / 因子 | ❌ 不可 |
| 实盘对齐模式 | 策略评估 / 参数校验 / WF / 样本外 | ✅ 唯一可用 |

**标准绩效输出：**

分年度收益、累计收益、年化收益、年化波动、夏普比率、最大回撤、盈亏比、胜率、交易笔数、信号层指标（IC / RankIC / Bin 分层）。

**交付物：** 《回测引擎开发文档》+《交易规则对齐确认书》

**禁止：**
- 回测规则与实盘不一致
- 忽略交易成本与滑点
- 使用未来函数（信号计算不得依赖未来数据）

> **案例：交易成本不是可选项。** 在一个 5 万元小资金策略的回测中，最初漏掉了佣金"最低 5 元"规则。单笔交易 ~5000 元，万 1 佣金仅 0.5 元，但实际券商收 5 元——实际成本是回测的 10 倍。这个小 bug 导致回测年化收益虚高约 1.5-2 个百分点。在 V4.2 修复后，策略年化收益下调了近 2%——这不是策略变差了，是之前算错了。

### 四-A：回测前自动校验清单（SOP V2.2 新增）

**每次执行回测前，必须通过以下 5 项自动化检查。任一未通过 → 中止回测，修复后重跑。**

```python
# 1. 前视偏差：所有特征列不存在 shift(-N) 或 iloc[i+N]
for col in feature_columns:
    assert not any(col in line for line in grep('shift(-'), f"特征 {col} 含未来函数")

# 2. 数据对齐：标签(fwd_ret)与当日(change_pct)低相关（动量效应<0.15正常）
assert abs(np.corrcoef(df['fwd_ret'], df['change_pct'])[0,1]) < 0.15

# 3. 特征完整性：训练集特征非空率 > 80%
for col in feature_columns:
    coverage = df[train_mask][col].notna().mean()
    assert coverage > 0.8, f"{col} 训练集覆盖率仅 {coverage:.0%}"

# 4. 标签分布：正负样本比在 0.3~0.7 之间
pos_ratio = df[train_mask]['label'].mean()
assert 0.3 < pos_ratio < 0.7, f"标签失衡: {pos_ratio:.0%}"

# 5. 时间窗口：训练集最大日期 < 样本外最小日期
assert df[train_mask]['trade_date'].max() < df[oos_mask]['trade_date'].min()
```

**通过后记录：** `sanity_check.log` 含时间戳 + 5 项结果。

## 第五阶段：风控参数调优与过拟合防控

**目的：** 在训练集内优化风控参数，同时严格防控过拟合。

**参数调优硬性规则：**

1. 仅允许在**训练集**内调参
2. 同类型参数仅允许**单次调整**，完成即永久冻结
3. 候选参数数量 ≤ 2 组（每增加一组候选，参数组合呈指数增长——3 组候选 × 3 个参数 = 27 种组合，在训练集上总能找到一组"看起来最好"的，但那是过拟合。2 组是硬上限，逼你在调参前想清楚。）
4. 禁止为抹平某几年历史收益针对性修改参数
5. 所有候选参数和选定依据必须书面记录
6. 选定后代码注释标注参数来源

**调参流程：**

```
训练集内跑全量 WF 回测（多候选参数）
    ↓
选出综合表现最优参数（不只看单区间收益）
    ↓
写入《参数调参记录单》，永久冻结
    ↓
代码注释标注：参数值 + 选定区间 + 回退方案
```

**交付物：** 《参数调参记录单》+《WF 回测报告》

**禁止：**
- 使用样本外数据调参
- 反复迭代同一参数
- 网格遍历风控参数

> **案例：仓位加权不是 alpha。** 在一个策略中，团队尝试了仓位加权——对 LGBM 打分高的股票多配、低的少配。结果夏普从 1.81 提升到 1.89（+0.08），但最大回撤从 5.4% 放大到 8.2%（+2.8%）。收益增幅远小于风险增幅。核心原因是：仓位加权本质上是一个杠杆放大器——在模型预测正确时放大收益，预测错误时放大亏损。它没有增加任何预测能力，只是放大了赌注。结论：**如果一个调参让夏普微升但回撤大幅放大，它不是 alpha，是放大器。**

### 五-A：消融测试（SOP V2.2 新增）

**任何新增特征/过滤/参数，必须做消融测试验证独立贡献。**

```
基线回测（无该项）
    ↓
加入该项
    ↓
对比：OOS 年化 / 夏普 / 回撤 / 胜率
    ↓
Δ年化 > +2pp 或 Δ夏普 > +0.1 → 保留
否则 → 删除，不保留
```

**示例：** 新增 ma_dev 特征 → 先跑 V4 12特征回测（基线） → 再跑 V5 10特征回测（含 ma_dev） → 对比改善 → 达标保留。

**消融记录写入：** `ablation_log.md`，每项含基线/加入后指标对比 + 是否保留决策。

## 第六阶段：样本外盲测与全维度验收

**目的：** 验证策略在未见过的数据上的泛化能力。

**盲测流程：**

1. 锁定全部参数（模型 + 风控 + 交易规则）
2. **一次性**运行封存的样本外数据
3. **仅查看三大全局指标**：年化收益、夏普、最大回撤
4. **禁止拆分查看单年收益**
5. 对比《策略立项单》中的硬性目标，逐项核验

**全维度验收清单：**

| 验收项 | 方法 |
|---|---|
| 未来函数排查 | 逐段代码审查信号计算时点 |
| 净度曲线平滑性 | 可视化检查，无极端回撤 |
| 结构性 alpha 底牌 | 随机信号替代模型，确认基础规则不产生伪 alpha |
| 极端行情压力测试 | 单独检验历史熊市、深度调整、震荡年份表现 |
| 交易逻辑抽查 | 随机抽取 N 笔历史交易，人工核对买卖时点、价格、止盈止损逻辑 |
| 数据集完整性 | 确认训练 / 样本外无交叉污染 |

**验收结论：**
- 全部达标 → 通过，进入模拟盘
- 核心指标不达标 → 参数回退至上一版本，放弃本次修改，重新研发

**交付物：** 《样本外盲测报告》+《策略上线验收报告》

**禁止：**
- 看完样本外结果后回头修改任何参数
- 拆分查看样本外单年收益

> **案例：归因分析能救命。** 一个双策略系统（老鼠潜伏 + 尾盘 T+1），在考虑是否砍掉老鼠策略时，没有凭直觉做决定，而是做了归因分析：尾盘 T+1 贡献了 91% 的收益，老鼠仅贡献 9%。但砍掉老鼠后整体收益反而下降——因为两个策略的选股域互补，老鼠覆盖了尾盘 T+1 不碰的股票类型。**如果只看净值曲线，你会觉得老鼠没用；但归因分析告诉你，它不是 alpha 来源，而是分散化工具。**

## 第七阶段：上线分层试运行

**目的：** 验证真实市场环境，逐步过渡实盘。

**三阶段推进：**

| 阶段 | 时长 | 资金比例 | 任务 |
|---|---|---|---|
| 模拟盘 | ≥ 2 周 | 0% | 验证链路（数据 → 信号 → 买卖 → 记账）零报错。2 周是最低要求——一周不够，因为涨跌停封死、停牌复牌、数据缺失等边界情况只在特定市场环境下才暴露，需要至少覆盖两个完整交易周。 |
| 小资金实盘 | ≥ 1 个月 | ≤ 10% | 对比回测 / 模拟盘 / 实盘绩效，记录现实损耗 |
| 梯度扩容 | 每次 ≥ 1 个月 | 30% → 60% → 100% | 监控容量与滑点变化 |

**模拟盘必须验证：**

- 买 / 卖撮合逻辑与回测一致
- 数据及时性（当日数据是否可获取）
- 异常处理（涨跌停、停牌、数据缺失）
- 净值计算准确

**交付物：** 《模拟盘运行报告》+《实盘交易日志》

**禁止：**
- 实盘规则与回测不一致
- 一次性满仓投入

### 七-A：实盘心理纪律

个人量化没有机构风控部门盯着你，最大的风险往往不是策略本身，而是你自己的行为。以下几条是实盘运行的底线：

| # | 纪律 | 为什么 |
|---|------|--------|
| 1 | **连续盈利不放大仓位** | 赚了钱容易飘，觉得"我行了"，然后加倍下注——这正是大亏的前奏。仓位只按策略规则执行，不因近期盈亏手动调整 |
| 2 | **连续亏损不恐慌砍仓** | 亏了钱容易慌，该下单时不敢下，或者手动止损。只要策略没触发风控红线，就按信号执行——回测里经历过更差的时期 |
| 3 | **盯盘不动手** | 盘中看到涨跌，手痒想"优化一下"。策略信号是收盘前生成的，盘中任何操作都是冲动交易，不在策略框架内 |
| 4 | **亏了不换策略，赚了不迷信策略** | 新手最容易犯的错：这周亏了就觉得策略不行，下周换一个。策略需要足够长的运行周期才能评价——至少一个季度。反过来，连续赚钱也不代表策略完美，可能是运气好 |
| 5 | **每周只看一次净值** | 每天看净值只会制造焦虑，导致频繁干预。定好每周固定时间看一眼，其他时间让策略自己跑 |

> **个人量化的最大优势也是最大劣势：没有上级管你。** 机构交易员有风控部门强制平仓，有合规部门审查操作。你什么都没有——唯一的风控是你自己。这几条纪律不是"建议"，是"如果你不想亏光就别违反"。

> **案例：模拟盘是最后的防线。** 一个尾盘 T+1 策略在回测上表现优异（夏普 1.81），但模拟盘上线第一周就暴露出两个回测中不存在的问题：（1）模拟盘引擎的仓位计算公式缺少 `stop_mult` 因子，导致实际下单仓位比回测多了 2 个百分点，回测和模拟盘跑的不是同一个策略；（2）佣金计算漏掉了"最低 5 元"规则，5 万元小资金账户中，实际佣金是回测估算的 10 倍。这两个 bug 在模拟盘运行近一周后才被发现——期间所有绩效数据都不具备参考价值。**如果跳过模拟盘直接上实盘，这两个 bug 的代价就是真金白银。** 模拟盘至少跑满 2 周这个数字，来源于这次经验：一周不够，因为有些问题只在特定市场环境下才暴露（如涨跌停封死、停牌复牌）。

## 第八阶段：版本管理与迭代规范

**版本编号规则：**

| 类型 | 编号 | 示例 | 变更范围 |
|---|---|---|---|
| 大版本 | V2.0 | 新增因子 / 换模型架构 | 必须重新走完整 SOP |
| 小版本 | V2.1 | 仅修改风控参数 / 交易成本 | 第五阶段起重新验证 |

**每个版本必须包含：**

- 策略代码（git tag）
- 模型文件
- 参数配置（含调参记录）
- 回测报告
- 验收报告

**迭代规则：**

任何迭代必须按 **训练集验证 → 样本外盲测 → 模拟盘** 的最小闭环执行，不允许跳过任何环节。

**版本回退：**

如果新版本上线后发现 bug 或绩效显著劣于预期，回退流程如下：

1. 立即回退到上一个稳定版本（git tag 直接切换 + 模型文件回退）
2. 记录回退原因和时间，写入版本日志
3. 在新版本修复 bug 后，重新走第五阶段起的验证流程
4. 禁止"边跑边改"——模拟盘/实盘运行期间不允许修改策略代码

**并行版本管理：**

当多个策略并行运行时，每个策略独立版本号，互不干扰：

| 策略 | 当前版本 | 状态 |
|------|----------|------|
| 策略 A | V2.1 | 实盘运行中 |
| 策略 B | V1.0 | 模拟盘验证中 |
| 策略 C | V3.0 | 开发中（回测阶段） |

> **实际经验：** 在一个双策略系统中，老鼠潜伏策略和尾盘 T+1 策略各自独立迭代，互不阻塞。如果因为一个策略的改动要求另一个策略也重新验证，会大幅拖慢迭代速度。每个策略的版本号、模型文件、回测报告独立管理，只在组合层面做归因分析。

## 过拟合防控 10 条红线

| # | 红线 |
|---|------|
| 1 | 禁止使用样本外数据训练模型、调参或优化规则 |
| 2 | 禁止反复迭代修改同一参数；同类型参数仅允许单次调整 |
| 3 | 禁止大规模网格遍历参数；候选参数 ≤ 2 组 |
| 4 | 禁止为抹平特定年份历史收益而针对性修改参数 |
| 5 | 禁止拆分查看样本外单年收益；仅允许查看全局指标 |
| 6 | 禁止回测规则与实盘操作不一致 |
| 7 | 禁止忽略交易成本、滑点、涨跌停等现实约束 |
| 8 | 禁止使用未来函数；信号计算不得依赖未来数据 |
| 9 | 禁止追求完美回测曲线；接受策略天然的年度收益分化 |
| 10 | 未经样本外盲测验收，禁止上线实盘 |

## 附录：失败案例集

> 以下案例均来自真实项目。它们不是"最佳实践"，是"活下来的实践"——每个案例都代表了一段被浪费的时间、一笔被高估的收益、或一个被推翻的假设。

### 新手常见行为错误

以下错误来自大量个人量化新手的真实踩坑经历。它们不是技术问题——而是**行为模式和认知偏差**。技术问题有标准答案，行为问题只有自己踩过才信。

#### 错误 1：回测赚钱就急着上实盘

**典型表现：** 写了个均线金叉策略，回测年化 30%，兴奋得睡不着，第二天就充钱开干。

**为什么是坑：** 回测赚钱 ≠ 实盘赚钱。至少有三个原因：（1）回测可能漏了手续费、滑点、涨跌停导致的无法成交；（2）回测用的是历史数据，你的策略可能只是"记住了"历史，碰到新行情就失效；（3）回测没有心理压力，实盘亏 5% 和回测亏 5% 是完全不同的体验。

**正确做法：** 回测通过后，至少跑 2 周模拟盘，确认链路无 bug。然后小资金（≤10%）跑 1 个月，对比回测和实盘数据。最后才梯度扩容。

#### 错误 2：这周亏了就换策略

**典型表现：** 策略实盘第一周亏了 3%，心想"这策略不行"，换一个新策略。第二周新策略亏了 2%，继续换。三个月换了六七个策略，没一个坚持超过两周。

**为什么是坑：** 任何策略都有连续亏损的时期——回测里也经历过。LGBM 策略在训练集上曾连续亏损 3 周，但不影响全年盈利。你每换一次策略，就中断了一次"让策略证明自己"的机会。

**正确做法：** 策略上线前设定一个"观察期"（至少一个季度），只在触发风控红线（如回撤超过立项时设定的最大值）时才考虑停掉。日常波动不干预。

#### 错误 3：回测太好了，反复调参把曲线调完美

**典型表现：** 回测夏普 1.5，觉得"还能更好"。调一调参数，1.8。再调一调，2.2。曲线漂亮得像教科书。信心满满上实盘，亏成狗。

**为什么是坑：** 这本质上是在拟合历史噪声。参数调得越多，回测曲线越好看，但策略在未来的表现越差。这就是本 SOP 反复强调的"过拟合"。

**正确做法：** 参数只调一次，调完永久冻结。候选参数不超过 2 组。如果回测曲线"太完美"，保持警惕——真实策略的回测曲线一定是有波动的。

#### 错误 4：满仓一只票

**典型表现：** 策略选出了"最强"的一只股票，觉得"这把稳了"，全部资金买入。

**为什么是坑：** 单票满仓意味着一只股票的黑天鹅（业绩暴雷、停牌、一字跌停）就能让你亏光。A 股历史上，看似"最稳"的白马股暴雷的案例并不少见。

**正确做法：** 单票仓位有上限（建议 ≤25%），总仓位有上限（≤50%），剩余资金配国债 ETF 或逆回购。仓位管理不是"限制赚钱"，是"防止亏光后没机会再赚"。

#### 错误 5：忽略手续费，以为小钱无所谓

**典型表现：** 5 万元本金，每天买卖一次，觉得万 1 的佣金就是 5 块钱，不值一提。

**为什么是坑：** 佣金最低 5 元 + 印花税万 5（卖出）意味着：买卖一次的固定成本至少 10 元。一年 240 个交易日，每天做一次买卖 = 2400 元成本，相当于本金的 4.8%。如果你的策略年化收益才 10%，手续费吃掉了一半。本金越小，手续费占比越高。

**正确做法：** 回测时把交易成本（佣金、印花税、滑点）全部算进去。小资金减少交易频率，不要做日内高频。如果策略年化收益 10%，全年交易成本超过 2% 就要警惕。

### 技术案例：可转债双低策略——全链路搭建，最终判死刑

**做了什么：** 从 Tushare 拉取 1126 只可转债的基本信息 + 25 万条日线数据，计算溢价率，按"价格 z-score + 溢价率 z-score"排名取 Top 20，日频轮动。

**结果：**

| 窗口 | 年化 | 最大回撤 | 夏普 |
|------|------|----------|------|
| 训练 2022-2023 | -13.03% | 26.6% | -1.14 |
| OOS 2024-2026 | -1.94% | 20.1% | -0.08 |

**失败根因：**
1. **估值陷阱**：便宜的可转债便宜是有原因的——市场定价是有效的。低价+低溢价通常意味着正股基本面恶化或信用风险。
2. **缺少关键过滤**：没有做信用评级过滤、剩余期限过滤、强赎状态过滤。
3. **2022 年流动性危机**：债市下跌引发理财产品赎回潮，转债被无差别抛售，双低策略变成"接飞刀"。
4. **2024-2026 年牛市**：正股大涨 → 溢价率大幅收窄 → 双低标的被"挤压"在低收益区，策略的防御性变成了踏空。

**教训：** 便宜不等于好。在加任何过滤维度之前，先问"这个维度是否区分了'打折的好货'和'该扔的垃圾'"。

### 案例 2：仓位加权——夏普微升，回撤暴增

**做了什么：** 对 LGBM 打分高的股票多配仓位、打分低的少配仓位。

**结果：**

| 指标 | 等权 | 仓位加权 | Δ |
|------|------|----------|-----|
| 夏普 | 1.81 | 1.89 | +0.08 |
| 最大回撤 | 5.4% | 8.2% | +2.8% |

**失败根因：** 仓位加权本质上是一个杠杆放大器——在模型预测正确时放大收益，预测错误时放大亏损。它没有增加任何预测能力，只是放大了赌注。

**教训：** 如果一个调参让夏普微升但回撤大幅放大，它不是 alpha，是放大器。放弃它。

### 案例 3：V5 特征扩展——AUC 只涨了 0.0089

**做了什么：** 将 LGBM 特征从 12 个扩展到 14 个，包括新增周频特征和行业特征。

**结果：** AUC 从基线提升 **+0.0089**。几乎可以忽略。

**失败根因：** 新增特征虽然通过了 IC 检验，但和已有特征高度相关，没有提供独立信息增量。

**教训：** 特征数量不是越多越好。增量收益必须覆盖代码复杂度、数据维护成本和过拟合风险。+0.0089 AUC 不值得增加两个特征。

### 案例 4：数据源切换——复权口径不一致导致回测幻觉

**做了什么：** 从 Tushare 前复权（qfq）切换到 QMT 后复权（hfq）。

**结果：** Tushare 回测夏普 1.8 的策略，QMT 模拟盘跑三天就亏了 2%。排查两天定位到是复权因子不一致导致的历史信号幻觉。

**失败根因：**
1. 前复权每次增量更新后历史价格漂移，回测结果不可复现。
2. 不同数据源的复权基准不同，混用等于在不同坐标系下做计算。

**教训：** 全程使用同一数据源、同一复权口径。如果要切换，必须全量重建数据库并重新跑一遍完整回测。
